Agents and Deployment

The first four chapters built skills for querying data. AI answered questions, generated charts, and wrote memos. But in each case, you were the orchestrator — you decided what to ask next, when to generate a chart, and when to synthesize.

This chapter introduces agents: AI systems that can plan, execute actions, observe results, and iterate autonomously. It also covers the practical path from a prototype on your laptop to a deployed tool on your corporate intranet.

What is an agent?

An agent is an AI system that pursues a goal through a loop of planning, executing, observing, and iterating. The distinction from a chatbot is autonomy. A chatbot handles a single turn: you ask, it answers, and you decide what to do next. An agent handles multiple steps: it breaks the task down, calls tools in sequence, checks the results, and adjusts its approach — all from a single instruction.

A research agent might search for information, read the results, summarize findings, and then search deeper based on what it learned. A data pipeline agent might query a database, clean the data, generate charts, write a report, and save everything to a folder. A coding agent might read requirements, write code, run tests, and fix failures.

The skills you built in Chapter 4 are the building blocks. A skill answers one question. An agent chains skills and other tools together to complete a workflow.

The agent loop

Every agent follows the same four-step loop. First, it plans by breaking the task into steps. Second, it executes by calling tools and running code. Third, it observes by checking the results of each action. Fourth, it iterates by adjusting its plan based on what it observed. Then the loop repeats until the task is complete.

This is exactly what your coding agent does when you give it a complex instruction. Ask it to “analyze the Chinook database, create three charts, and write a summary report,” and it will plan the steps, execute them in sequence, check each result, and produce the final deliverable. You have been using an agent since Chapter 2 — you just did not call it that.

How agents use tools

The AI does not just generate text. It decides which tool to call, formats the request, and interprets the result.

Tools can be API calls (REST endpoints, webhooks, external services), file operations (reading, writing, and organizing files on disk), or database queries (SQL via the skills from Chapter 4). The tool-use protocol works the same way regardless of the tool: the AI receives a goal from the user, selects the right tool, formats the call, and interprets the result. Then it decides whether the task is complete or whether another tool call is needed.

Consider a concrete example. You give the agent a single prompt: “Query the Chinook database for the top ten countries by revenue. Create a bar chart and a summary table. Write a one-page PDF report with the chart, table, and a three-sentence executive summary. Save everything to the reports folder.”

The agent queries the database using the Chapter 4 skill, generates a matplotlib chart and saves it as a PNG, builds a PDF with the chart, table, and narrative, and saves all files to the specified folder. Four tools, one prompt. The agent orchestrated the entire pipeline.

Agent architectures

Not all agents work the same way. The architecture depends on the complexity and stakes of the task.

A single-agent architecture uses one AI with access to multiple tools. This is what Claude Code, Gemini CLI, and Codex each provide, and it is sufficient for most workflows.

A multi-agent architecture uses multiple specialized agents that collaborate. One agent might plan the workflow while others execute individual steps. This is useful for complex pipelines where different steps require different expertise.

A human-in-the-loop architecture adds approval checkpoints. The agent pauses at predefined points and waits for human confirmation before continuing. This is essential for high-stakes actions where mistakes are costly or irreversible.

The permission model

The most important architectural decision is the permission model: what can the agent do without asking?

At the read-only level, the agent can query data and generate reports but cannot modify anything. A database analyst agent would operate at this level.

At the write-local level, the agent can save files to a project folder. A report generator that creates PDFs and charts would operate here.

At the write-external level, the agent can send emails, update systems, and interact with external services. Workflow automation agents operate here and require the most careful governance.

Start with read-only. Add permissions as trust builds. The trust spectrum from Chapter 7 applies directly: the higher the stakes, the more checkpoints.

Containers and sandboxed execution

When AI runs code, you need to control where it runs. On your laptop during development, a coding agent runs in your environment with access to your files. That is fine for prototyping. In production, you want isolation.

Built-in sandboxing

Codex includes OS-level sandboxing by default — it runs generated code in an isolated environment even on your laptop. Claude Code and Gemini CLI rely on permission prompts to control what the agent can do, but do not sandbox execution. For production deployment, all three benefit from the container-based isolation described below.

A container (Docker is the standard) is a disposable, isolated computing environment. Think of it as a virtual computer that exists only for one task and is destroyed afterward. It can only access what you explicitly allow. A bug in the AI-generated code cannot affect your other files, your database, or your network.

The sandbox pattern for production looks like this: a user asks a question through a web interface, the AI agent plans and orchestrates, the agent runs code inside a sandboxed Python container, the container queries the database through a read-only connection, and the results flow back to the user.

The user never touches the database. The agent never escapes the sandbox. This is defense in depth.

Deploying to a corporate intranet

Deploying an AI data tool on your intranet requires two contributions.

You provide the SKILL.md from Chapter 4 — your domain knowledge — along with test queries, expected results, and business rules.

IT provides a web wrapper (a browser interface built with Streamlit, Flask, or an internal portal), SSO authentication so only authorized users can query, and logging so every query is recorded for compliance.

The skill file is the deployment artifact. IT adds infrastructure; you provide domain knowledge.

The deployment path

The path from prototype to production follows three phases.

In the sandbox phase, you build and test the skill on your laptop. This is what you did in Chapters 2 through 4.

In the audit phase, you red-team the skill, review the SQL it generates, and check for data leakage. This is covered in Chapter 7.

In the deploy phase, IT wraps the skill in a web interface with SSO, logging, and read-only credentials. The skill you built in Chapter 4 is the same artifact that powers the deployed tool. The difference is infrastructure, not intelligence.

Sandboxed Python on the intranet

The end state is straightforward: anyone on your team submits a question through a web interface, the agent runs code in a container, and the answer comes back. Users see a simple chat interface with no terminal and no code. The agent uses your SKILL.md to query the database correctly. All queries are logged, the database is read-only, and the container is disposable.

The technology exists today. The bottleneck is organizational readiness — governance, trust, and the willingness to start.

Exercises

Exercise 1: Multi-Step Agent Workflow

Open your coding agent in a folder containing the Chinook database and your SKILL.md from Chapter 4. Give it a single compound instruction that requires at least three steps.

For example: “Query the Chinook database for the top ten artists by revenue. Create a bar chart. Write a one-paragraph summary. Save the chart and summary to a reports folder.”

Observe the agent loop. How many distinct tool calls does the agent make? Does it check its own work? If the output is not right, refine your prompt and try again.

The deliverable is your multi-step prompt, the number of tool calls you observed, and the final output.

Exercise 2: Permission Boundary Design

Pick a use case from your organization: a sales report agent, a customer support analyzer, or an inventory monitor.

List five actions the agent would need to take. Classify each as auto-approve (the agent can do it without asking), require-human-approval (the agent pauses and waits), or never-allow (the agent must not do this under any circumstances).

Discuss with a colleague: would you trust this agent with these permissions? What would need to change for you to feel comfortable?

The deliverable is your five-action permission matrix with classifications.

Exercise 3: Workflow Decomposition

Consider the task: “Prepare the quarterly business review for the CEO.”

List the five to seven steps a human analyst would take to complete this task. For each step, identify what tool the agent would use and what data it would need. Then write a single prompt that describes the full workflow.

Which steps need human approval before the agent continues? Where are the highest-risk points?

The deliverable is your step-by-step decomposition, the single-prompt version, and notes on which steps require human oversight.

Exercise 4: Agent Use Case Inventory

Review your team’s recurring tasks and identify five that are multi-step, repetitive, and data-driven — good candidates for agent automation.

For each, describe the trigger (what starts the task), the steps (what happens), and the output (what gets delivered). Rate each on automation potential (high, medium, or low) and risk level (high, medium, or low).

Pick the highest-potential, lowest-risk workflow as your pilot candidate and write one paragraph explaining why you would start there.

Exercise 5: Deployment Architecture Sketch

Design the deployment architecture for your Chapter 4 skill.

Draw a simple diagram showing the flow from user to web interface to agent to database. Annotate where SSO happens, where the sandbox boundary is, and where queries are logged.

List three questions you would need to answer before IT would approve the deployment. Then write a one-paragraph pitch explaining why IT should invest in building this.

The deliverable is the architecture diagram and the pitch paragraph.
