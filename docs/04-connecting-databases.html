4  Connecting Databases

Module 1 described the vision: replacing dashboards with natural-language queries. This module makes it real. You will connect a coding agent to a database and build the tools that let anyone query it in plain English — no SQL required.

The first two modules worked with individual files — a CSV placed in a folder, a memo pasted into a conversation. Most real business data does not live in spreadsheets. It lives in databases: structured collections of related tables, hosted on servers, queried with SQL.

This module connects AI to those databases. By the end, you will understand how your organization can create tools so that everyone can query a database in plain English — no SQL required. As throughout this course, the examples use Claude Code, but the same workflows apply to other coding agents (see Appendix).

4.1 From spreadsheets to databases

In Module 2 you analyzed a spreadsheet — a single table of rows and columns living in a file on your machine. Most real business data lives in databases: many related tables linked by keys, hosted on a server, queried with SQL.

SQL is the language databases understand. You do not need to learn it. AI writes the SQL for you. But you do need to tell the AI what is in the database — which tables exist, what the columns mean, and how the tables relate to each other.

4.2 How AI connects to a database

There are several ways to give AI access to your data, ranging from simple to sophisticated.

Skill files. The simplest approach: write a markdown file that describes your database schema and hand it to a coding agent. The agent reads the description, writes SQL, and executes it. No special infrastructure is needed — just a file and a coding agent that can run code. This is what we build in this module.

Direct API calls. A lightweight application — even fifty lines of Python — can send a user’s question and a schema description to an LLM, get back SQL, run it, and display the results. This is the pattern behind the demo app at the end of this module. It works for prototypes and internal tools where you control the deployment.

MCP (Model Context Protocol). An open standard that lets AI connect to external tools — databases, CRMs, calendars, anything with an API. Think of it as USB for AI. The AI assistant acts as an MCP client and connects to MCP servers that wrap external systems. MCP is most useful when you need to connect to many different tools in a single conversation or when IT wants a standardized way to manage AI’s access to corporate systems.

Vendor platforms. Salesforce, Snowflake, Databricks, and others are embedding natural-language query directly into their products. If your data already lives in one of these platforms, you may not need to build anything — just enable the feature.

All of these approaches share the same core idea: give the AI a description of your data and let it write the query. The differences are in how the description is delivered, who runs the query, and how much infrastructure is involved. We start with skill files because they require no setup and teach the underlying pattern that every other approach builds on.

4.3 The sample database: Chinook

The exercises in this module use the Chinook sample database, which you can download here: chinook.db. It represents a music retailer’s operations and contains 11 related tables.

The key tables are employees (8 staff members), customers (59 customers across 24 countries), invoices and invoice_items (412 transactions), and tracks, albums, artists, and genres (a product catalog of more than 3,500 tracks). It is a publicly available SQLite file that requires no setup — just place it in a folder and point your coding agent at it. The same skill pattern you learn here works with any database.

4.4 What is a skill?

Out of the box, the coding agent can read files, write code, and run it. But you can extend what it knows how to do by installing skills — packaged instructions that teach the agent a specific workflow.

Claude Code has a skill system built in. A skill is a folder containing a markdown file that describes the task, the steps, and the output format. When you invoke a skill, Claude Code reads the instructions and follows them. The installer from the preface already set up four skills that are particularly useful for business work:

  • xlsx — read, create, and edit Excel spreadsheets with formulas, formatting, and charts.
  • docx — create and edit Word documents with tracked changes, comments, and formatting.
  • pptx — create and edit PowerPoint presentations.
  • skill-creator — a guide for building your own skills. You will use this later in this module when you create a database skill.

You can verify the skills are installed by opening the Code tab and typing / — the available skills appear as slash commands.

A skill is a set of instructions that specializes a general-purpose AI for a specific task. Every AI customization works the same way: additional text is added to the prompt. A skill provides three things: domain knowledge (what the data means), workflow instructions (how to do the task), and output format specifications (how to present results).

Skills are not limited to databases. You could build a skill for weekly report generation, customer support analysis, or any recurring task where AI benefits from domain context.

4.5 The SKILL.md file

The skill is a markdown file that the coding agent reads when you invoke it. The structure has three parts.

The first part is the description and location: what the skill does and the path to the database file.

The second part is the schema and relationships: every table, its columns, data types, and how tables link to each other. This is the most important part. The more precisely you describe the schema, the better the AI’s queries will be. Include column descriptions (“status is one of: active, completed, on-hold”), explain join paths (“invoices link to customers via CustomerId”), and note any business logic (“revenue means invoice_items.UnitPrice times Quantity”).

The third part is the instructions: query style, output format, and chart preferences. For example: “Always show the SQL query before showing results. Use Plotly for interactive charts. Annotate the highest and lowest values.”

The skill file is just text. Anyone can edit it. No programming is required — just clear descriptions of your data.

4.6 Creating the skill step by step

Create a project folder with the SQLite database file and a file called SKILL.md. Write the schema, describing each table, its columns, and its relationships. Then test by asking questions and refining based on what Claude gets wrong.

Your first draft will not be perfect, and that is fine. The skill improves through iteration. Every time Claude misinterprets a question or uses the wrong join path, you add a clarification to the SKILL.md. After a few rounds of testing, the skill handles the most common questions reliably.

You do not have to write the skill from scratch. Point your coding agent at the database and ask it to generate the SKILL.md for you. It will inspect the tables, columns, and foreign keys and produce a first draft. Your job is to read the result, verify that the descriptions match your understanding of the data, and add any business context the agent could not infer on its own.

4.7 Building the Chinook skill

To make this concrete, consider a scenario. You are an operations director at a music retailer. The CEO wants a recommendation by Friday: which markets to invest in, which product lines to promote, and how to improve per-customer revenue.

The data lives in the Chinook database. The answer requires joining customer, invoice, and catalog data across multiple tables. There is no time to wait for an analyst.

The solution: build a skill that knows the Chinook schema, then ask questions in plain English. The agent writes SQL, runs it, and returns charts.

4.8 Decision-driven queries

With the skill active, every question should connect to the CEO’s decision.

“Top five countries by revenue, bar chart, sorted” produces a bar chart comparing categories. “Monthly revenue trend line — is the business growing?” produces a line chart showing trends over time. “Plot average order value versus number of orders by country” produces a scatter plot that reveals clusters of high-value versus high-frequency markets.

You can go deeper. “Correlation heatmap of all numeric columns” shows which metrics move together. “Make that chart interactive so I can hover for details” switches from matplotlib to Plotly, saving the result as an HTML file. “Annotate the outlier country on the scatter plot” adds context to the visualization.

Each question is a different lens on the same decision. The skill ensures that every query uses the correct tables, joins, and business logic.

4.9 SQL transparency

One of the most important features of this workflow is that you can always see the SQL. When you ask “Which country generates the most revenue? Show a bar chart of the top 10,” Claude generates a SQL query that selects country and summed revenue from the invoices and customers tables, groups by country, orders by revenue descending, and limits to ten results.

You can read the SQL and verify that the joins are correct, the filters make sense, and the aggregation matches your intent. Analysts can review the logic. This builds trust: AI is not a black box.

If you ask a follow-up — “Break down the US revenue by state” — Claude writes a new query with a WHERE clause and additional grouping. The SQL is always available for inspection.

4.10 Iterating on the skill

Common refinements include adding column descriptions, specifying default sorting and format rules, and setting chart preferences. You can specify whether charts should be static (matplotlib, good for reports and slides) or interactive (Plotly, with hover, zoom, and filter, saved as HTML files and shareable via email). The same prompt works for both — just add “interactive” or “Plotly.”

Skills turn institutional knowledge into executable instructions. Instead of a procedures manual that sits on a shelf, you have a prompt that AI follows every time. The knowledge is not just documented; it is operationalized.

4.11 Behind the firewall

Understanding the data flow matters for security. When you use a coding agent, the flow works like this: you type a question, the agent sends your question and the schema description to the LLM (hosted in the cloud), the LLM returns SQL, and the agent executes that SQL against your database locally. The raw data never leaves your network — only your question and the schema travel to the cloud. The query results stay on your machine.

In a deployed app (like the Streamlit demo), the app replaces the coding agent: the app sends the question to the LLM, receives SQL back, runs it against the database, and displays results to the user. Again, the LLM never sees the actual data — only the question and the schema.

IT controls which databases the agent or app can access and can restrict connections to read-only credentials.

For organizations that need to connect AI to multiple systems — databases, CRMs, financial data providers, project management tools — MCP provides a standardized way to manage those connections. The next section explains how it works.

In this module we connect to a local SQLite file. In your organization, the same pattern connects to PostgreSQL, Snowflake, or any database your team uses.

4.12 MCP: connecting to everything else

The skill-file approach works well when AI needs access to one database. But organizations often need AI to reach many systems at once — databases, CRMs, financial data providers, project management tools. This is what MCP (Model Context Protocol) is for.

MCP is an open standard — think of it as USB for AI. A coding agent acts as an MCP client. Each external system is wrapped in an MCP server — a small program that advertises the tools it offers (query a database, look up a customer, fetch a stock price). When you connect an MCP server, its tools appear automatically in the agent’s toolkit. The agent decides which tools to call based on your question. You can connect multiple servers at once, and the agent sees all of their tools in a single conversation.

Each MCP server handles authentication, rate limiting, and data formatting so the agent does not have to. The agent never sees API keys or raw API responses — the server manages those details and returns clean, structured results.

In Claude Desktop, MCP servers are configured in a JSON settings file or installed as one-click desktop extensions. Once configured, a server’s tools are available every time you open the Code tab.

Corporate MCP servers

Major platforms and data providers now offer MCP servers. The list is growing quickly. Here is a sample of what is available:

Databases and data platforms — Snowflake, Databricks (Unity Catalog, Vector Search, Genie), Google BigQuery, PostgreSQL, MongoDB.

Financial data — FactSet (fundamentals, M&A, supply chain), Bloomberg (internal adoption), LSEG/Refinitiv, Moody’s, S&P Global, Nasdaq, Alpha Vantage (stock prices, options, forex, economic indicators), Financial Modeling Prep (financial statements, SEC filings).

CRM and ERP — Salesforce (Agentforce), HubSpot, Microsoft Dynamics 365, SAP.

Business intelligence — Power BI (query semantic models, create measures and relationships).

Cloud platforms — AWS (60+ servers spanning infrastructure, Lambda, analytics), Azure (40+ servers), Google Cloud.

Collaboration — GitHub, Slack, Notion, Google Drive, Asana, Jira, Linear.

Payments — Stripe, PayPal.

Universal connectors — CData Connect AI (350+ enterprise data sources through a single MCP interface), Workato (100+ pre-built servers).

The MCP registry at registry.modelcontextprotocol.io lists thousands of public servers, and enterprises can run private registries for internal tools.

4.13 The skill as deployment artifact

A polished skill is what gets deployed on your intranet. The skill file encodes which tables matter, how they relate, and what the business terminology means. IT adds the infrastructure: SSO authentication, read-only database credentials, and query logging for audit. Module 5 shows how to chain this skill into an automated reporting pipeline.

4.14 A natural-language query app

The skill-file approach works well with a coding agent, but what about the rest of your organization? Not everyone will use a terminal. The same pattern — send a question and schema to an LLM, get SQL back, run it — can be wrapped in a simple web app that anyone can use from a browser.

In production, this app would be deployed on the corporate intranet, the database would be a PostgreSQL or Snowflake instance on the company’s servers, and the LLM would be an enterprise-licensed model (more on licensing and compliance in Module 6). For this demo, we run everything locally: a small Streamlit app on your laptop, the Chinook SQLite file in the same folder, and Claude Sonnet via the Anthropic API.

Setting up the Anthropic API

The demo apps call Claude Sonnet through the Anthropic API. You will need an API key.

  1. Sign up at console.anthropic.com and add a payment method.

  2. Create an API key under Settings → API Keys.

  3. Set the key as an environment variable so the demo apps can find it:

    Mac / Linux — add to your shell profile (.bashrc or .zshrc) to persist:

    export ANTHROPIC_API_KEY="sk-ant-..."

    Windows — run once in a terminal, then restart the terminal:

    setx ANTHROPIC_API_KEY "sk-ant-..."

API calls are billed per token. A short request costs a fraction of a cent; a longer conversation might cost a few cents to a few dollars. Current prices are listed at anthropic.com/pricing.

Download the app files here: chinook-query.py and requirements.txt. Copy chinook.db into the same folder, install the dependencies by running pip install -r requirements.txt in a terminal or by asking your coding agent to install them, and then run:

streamlit run chinook-query.py

What the app sends to the LLM

When you type a question, the app sends two things to the LLM: a system prompt containing the database schema, and your question. Here is the system prompt (abbreviated):

You are a SQL assistant. Given the schema below and a user
question, return ONLY a single SQLite SELECT statement.

The Chinook database has these tables:

artists(ArtistId, Name)
albums(AlbumId, Title, ArtistId) → artists
tracks(TrackId, Name, AlbumId, GenreId, UnitPrice) → albums, genres
invoices(InvoiceId, CustomerId, InvoiceDate, Total) → customers
invoice_items(InvoiceLineId, InvoiceId, TrackId, UnitPrice, Quantity)
...

Revenue = SUM(invoice_items.UnitPrice * invoice_items.Quantity).

Notice what is not in this prompt: no actual data. The LLM sees only the table and column names — the same schema description you would put in a skill file.

What comes back

If you ask “What are the top 5 countries by revenue?”, the LLM returns pure SQL:

SELECT c.Country, SUM(ii.UnitPrice * ii.Quantity) AS Revenue
FROM invoice_items ii
JOIN invoices i ON ii.InvoiceId = i.InvoiceId
JOIN customers c ON i.CustomerId = c.CustomerId
GROUP BY c.Country
ORDER BY Revenue DESC
LIMIT 5;

The app runs this query against the local chinook.db file and displays the results in a table. The generated SQL is shown above the results — the same transparency principle discussed in the “SQL transparency” section. The LLM never sees the query results; it only writes the SQL.

The data flow

The complete round trip looks like this:

  1. You type a question in the browser.
  2. The app sends your question + the schema description to the LLM (cloud).
  3. The LLM returns a SQL query (cloud → your machine).
  4. The app executes the SQL against the database (entirely local).
  5. The app displays the results in your browser (entirely local).

The only things that leave your machine are the question and the schema. The actual data — customer names, revenue figures, transaction records — never goes to the cloud.

4.15 Adding charts and visualizations

The query app returns tables, but business questions often need charts. “Show me a bar chart of revenue by country” should produce a bar chart, not a grid of numbers. With a coding agent like Claude Code, you would get the chart directly — the agent writes Python, executes it, and displays the image. But a deployed web app cannot do that unless we change what the LLM returns.

The fix is straightforward: instead of asking the LLM for SQL, ask it for a short Python script that queries the database and produces a chart. The app executes that Python and displays whatever it creates — a matplotlib figure, a Plotly chart, a data table, or all three.

The changed system prompt

The original query app’s system prompt said: return ONLY a single SQLite SELECT statement. The chart app replaces that with:

You are a Python code assistant for the Chinook music database.
Given the schema below and a user question, return ONLY a
Python script.

Rules for the Python script:
- Use sqlite3 to query the database. The connection is
  already available as `conn`.
- Use pandas for data manipulation.
- If the question asks for a chart, use matplotlib.
  Call st.pyplot(fig) to display it.
- Always store query results in a DataFrame called `df`.

The schema description is identical — the LLM still sees only table and column names, never the actual data.

What comes back

If you ask “Bar chart of the top 5 countries by revenue,” the LLM now returns Python instead of SQL:

df = pd.read_sql_query("""
    SELECT c.Country, SUM(ii.UnitPrice * ii.Quantity) AS Revenue
    FROM invoice_items ii
    JOIN invoices i ON ii.InvoiceId = i.InvoiceId
    JOIN customers c ON i.CustomerId = c.CustomerId
    GROUP BY c.Country
    ORDER BY Revenue DESC
    LIMIT 5
""", conn)

fig, ax = plt.subplots()
ax.bar(df["Country"], df["Revenue"])
ax.set_ylabel("Revenue ($)")
ax.set_title("Top 5 Countries by Revenue")
st.pyplot(fig)

The SQL is still there — embedded inside a read_sql_query call — but now the script also creates the chart. The app executes this code and displays both the chart and the underlying data table.

The data flow is unchanged

The security model is the same as the SQL-only app. The LLM sees only the question and the schema. The Python executes locally against the local database. No data leaves your machine. The difference is that the app now executes Python rather than a single SQL statement, which gives the LLM more expressive power — it can sort, reshape, and visualize the data in ways that raw SQL cannot.

Download the chart app files here: chinook-charts.py and requirements.txt. Copy chinook.db into the same folder, install the dependencies by running pip install -r requirements.txt in a terminal or by asking your coding agent to install them, and then run:

streamlit run chinook-charts.py

A note on executing generated code

The chart app uses Python’s exec() to run LLM-generated code. For a local demo this is fine — you are running it on your own machine. In a production deployment, executing AI-generated code requires sandboxing. The sections below explain the production-ready alternative: tool-use agents with sandboxed execution.

From demo to production

This is a demo, not a production application. There is no authentication, no guardrails on the SQL, and no sandboxing of query execution. The following sections describe how to cross that gap — from a fifty-line prototype to something you can hand to IT.

4.16 Agents with Python tools

The chart app works by asking the LLM to generate arbitrary Python code and then running it with exec(). This is flexible — the LLM can write whatever it needs — but hard to control. The LLM could generate code that reads files, makes network requests, or runs indefinitely. And when something goes wrong, the error is somewhere inside a blob of generated code.

Tool use is the production alternative. Instead of generating arbitrary code, the LLM selects from a menu of Python functions you define and calls them by name. You control exactly what the agent can do. The LLM only decides which tool to call and what arguments to pass.

For the Chinook query app, the tool menu might look like this:

def run_query(sql: str) -> list[dict]:
    """Execute a SQL SELECT and return rows as a list of dicts."""
    cursor = conn.execute(sql)
    columns = [d[0] for d in cursor.description]
    return [dict(zip(columns, row)) for row in cursor.fetchall()]

def make_bar_chart(sql: str, x_col: str, y_col: str, title: str) -> str:
    """Run a query and produce a bar chart. Returns path to saved PNG."""
    df = pd.read_sql_query(sql, conn)
    fig, ax = plt.subplots()
    ax.bar(df[x_col], df[y_col])
    ax.set_title(title)
    path = f"/tmp/chart_{title.replace(' ', '_')}.png"
    fig.savefig(path)
    return path

def summarize_table(table_name: str) -> dict:
    """Return row count and column names for a table."""
    count = conn.execute(f"SELECT COUNT(*) FROM {table_name}").fetchone()[0]
    cursor = conn.execute(f"SELECT * FROM {table_name} LIMIT 0")
    columns = [d[0] for d in cursor.description]
    return {"table": table_name, "rows": count, "columns": columns}

You describe these functions to the LLM as tool definitions — name, description, and parameter types — and the LLM reads the descriptions to decide which tool fits the question. It never sees the implementation. The tool definitions are the API; the Python functions are the implementation behind it.

This gives you two things exec() cannot: a bounded set of operations (the agent can only call the tools you register) and a structured audit trail (every tool call is a named, logged event with specific inputs and outputs).

Tool use also enables the LLM to do multi-step reasoning within a single question. “Which sales rep should I assign this account to?” might trigger: run_query to get account revenue, run_query to get rep workloads, run_query to check rep industry specializations, then a final synthesis. The LLM orchestrates the sequence; your code just runs the tools it requests.

4.17 The agent loop

Tool use requires a conversation loop. The LLM does not call a tool directly — it returns a structured message saying “I want to call run_query with these arguments,” and your code actually executes the function and returns the result. Then the conversation continues until the LLM signals it is done.

The pattern:

messages = [{"role": "user", "content": question}]

while True:
    response = client.messages.create(
        model="claude-sonnet-4-6",
        system=system_prompt,
        tools=[t["schema"] for t in TOOLS.values()],
        messages=messages,
    )

    if response.stop_reason == "end_turn":
        return response.content[0].text       # done — return final answer

    # LLM wants to call a tool
    tool_use_block = next(b for b in response.content if b.type == "tool_use")
    result = TOOLS[tool_use_block.name]["fn"](**tool_use_block.input)

    # Append the LLM's tool request and the result to the conversation
    messages.append({"role": "assistant", "content": response.content})
    messages.append({"role": "user", "content": [{
        "type": "tool_result",
        "tool_use_id": tool_use_block.id,
        "content": str(result),
    }]})
    # Loop: LLM sees the result and decides what to do next

Each iteration the LLM either calls a tool or signals it is done.

This loop is what Claude Code runs when it executes a multi-step task. You can wrap it with timeouts (stop after N iterations), cost tracking (count tokens per request), and retry logic (re-run a failed tool call with a corrected argument) — all without changing the core pattern.

4.18 Containers and sandboxing

Tool use limits what the agent can request. Containers limit what the code can reach. Both are needed in any deployment that handles real data.

The risk is concrete. If your app runs LLM-generated code — via exec() or a code-execution tool — and the app is exposed to users, a crafted input could produce code that reads credentials from the filesystem, makes outbound network calls, or exhausts server memory. Sandboxing prevents all of this.

Docker. A container is an isolated process with its own filesystem and network stack. It is created from an image (a recipe specifying the OS, libraries, and files), runs the code, and is discarded. Nothing it does persists after exit.

A minimal sandbox for executing generated Python:

docker run \
  --rm \                          # discard container after exit
  --network none \                # no internet access
  --memory 256m \                 # memory cap
  --cpus 0.5 \                    # CPU cap
  --read-only \                   # immutable filesystem
  --tmpfs /tmp:size=64m \         # writable scratch space only
  python:3.12-slim \
  python -c "$generated_code"

Code runs inside the container and cannot touch your database credentials, cannot make API calls, and terminates at timeout. The database connection is the key exception: the run_query tool runs outside the container under read-only database credentials, passing results in as data. Untrusted computation happens inside the container; trusted data access happens outside. This separation is the standard pattern.

Hosted sandboxes. Running containers per-request adds operational work. Several services manage this:

  • E2B — a cloud service built for executing LLM-generated code. Each call spins up an isolated micro-VM, runs the code, and returns the result. No infrastructure to manage; the API is a few lines of Python.
  • Modal — serverless containers that scale to zero between requests. You define the Python environment as code; Modal handles cold starts, scaling, and isolation. Useful when the agent needs heavier compute or persistent state across calls.
  • Daytona / Morph — similar hosted execution environments with different tradeoffs on latency and cost.

For an intranet prototype, one Docker container per request is fine. For a multi-user deployment, a hosted sandbox service avoids significant infrastructure overhead.

4.19 Agent harnessing

A harness is the scaffolding that wraps an agent: the system prompt, tool registry, conversation loop, error handling, and logging. The demo apps have an implicit harness — a system prompt and exec() — but without the structure needed to hand off to IT.

A minimal production harness has four components.

Tool registry. A dictionary mapping tool names to Python functions and their JSON schema descriptions. Keep the registry separate from the loop so you can add or remove tools without touching control flow.

TOOLS = {
    "run_query": {
        "fn": run_query,
        "schema": {
            "name": "run_query",
            "description": "Execute a SQL SELECT and return rows as JSON.",
            "input_schema": {
                "type": "object",
                "properties": {"sql": {"type": "string",
                                       "description": "A valid SQLite SELECT statement."}},
                "required": ["sql"],
            },
        },
    },
    "make_bar_chart": { ... },
}

Conversation manager. Tool use requires maintaining conversation history — the LLM’s tool requests and your tool results must both appear in the message list. A conversation manager handles appending, truncation (messages grow long in multi-step tasks), and serialization for logging.

Error handling. Tools fail. SQL has syntax errors; the database connection drops; a chart library raises an exception. The harness catches these and returns a structured error to the LLM, which can retry with a corrected argument. Without error handling, a single failed tool call terminates the entire conversation.

try:
    result = TOOLS[tool_name]["fn"](**tool_input)
    content = json.dumps(result)
    is_error = False
except Exception as exc:
    content = f"Error: {exc}"
    is_error = True

messages.append({"role": "user", "content": [{
    "type": "tool_result",
    "tool_use_id": tool_use_id,
    "content": content,
    "is_error": is_error,
}]})

Observability. Log every tool call: which tool, what arguments, how long it ran, whether it succeeded. This gives you an audit trail (who asked what query, which SQL ran against the database), a debugging surface (why did the agent loop six times?), and cost data (how many tokens per question).

2026-03-23 14:02:11 | tool=run_query | sql="SELECT ..." | rows=12 | ms=43
2026-03-23 14:02:12 | tool=make_bar_chart | title="Top 5 Countries" | ok=True | ms=210
2026-03-23 14:02:13 | stop_reason=end_turn | input_tokens=1840 | output_tokens=312

A complete harness for the Chinook query app is around 150 lines of Python. The code is not complex, but it is what separates a demo from something IT can deploy: they can read exactly what tools exist, what the agent can access, and where the logs go. Module 6 covers the governance layer on top: authentication, data-processing agreements, and query approval workflows.

4.20 Exercises

Exercise 1: Build a Chinook Skill

Download chinook.db and place it in a new project folder. Create a SKILL.md file in the same folder with three sections: a description and database path, at least three table descriptions with columns and relationships, and three example questions.

Open your coding agent in the folder and test three decision-relevant queries. For each, ask Claude to show the SQL and verify that the joins and filters are correct. Then ask for an interactive Plotly chart and iterate on it at least once — change colors, add annotations, or improve the layout.

You should end up with a working SKILL.md and at least one interactive chart.

Exercise 2: Database Schema Detective

Connect your coding agent to the Chinook database without any SKILL.md — no documentation at all.

Ask: “What tables are in this database? How are they related?” Let Claude discover the schema on its own. Then ask: “If I were the CEO of this music company, what three questions should I be asking?”

Follow up with two questions that require joining at least two tables (for example, “Revenue by artist” requires joining invoices, invoice_items, tracks, albums, and artists).

Compare the agent’s schema discovery to what you know from the module. Did it miss anything?

Exercise 3: Decision Scenario Queries

You are the Chinook CEO. The board wants you to cut two of the five lowest-performing genres from the catalog to reduce licensing costs.

Using your Chinook skill (or a bare connection to chinook.db), answer these questions in sequence: Which genres contribute the least revenue? For those two genres, how many unique customers would be affected? Are any of those customers high-value customers who also buy other genres?

Write one sentence stating which two genres you would cut and why.

Exercise 4: Skill for Your Own Data

Take a CSV from your work (anonymize if needed) or a public dataset you care about. Ask the agent to convert it to a SQLite database. Then write a SKILL.md describing the tables, columns, relationships, and five example questions.

If you do not have a work dataset, try one of these public sources:

  • Kaggle Datasets — thousands of downloadable CSVs across business, finance, healthcare, sports, and more.
  • UCI Machine Learning Repository — classic datasets for analysis, including census, retail, and banking data.
  • data.gov — U.S. government open data on economics, education, health, and public safety.
  • World Bank Open Data — country-level economic and development indicators.

Test five queries and refine the skill until all five return correct results.

Exercise 5: The Monday Morning Report

Identify a recurring report your team produces — weekly sales summary, monthly KPIs, pipeline report. Write a SKILL.md that includes the database schema, the specific queries needed, and the output format.

Add an instruction to the skill: “When I say ‘Monday report,’ run these three queries and format the results as a one-page summary.” Test it. Does one command produce your entire report?

Exercise 6: Build Your Own Query App

Ask your coding agent to build a natural-language query app similar to the chinook-query demo, either from scratch or by improving the demo. Possible extensions: add chart generation for numeric results, show query history in a sidebar, add error handling that suggests corrections when the SQL fails, or let the user choose between different databases.

Exercise 7: Build a Tool-Use Agent

Ask your coding agent to convert the chinook-charts demo from an exec()-based app to a tool-use agent.

Define at least three tools: run_query(sql), make_bar_chart(sql, x_col, y_col, title), and one of your own choosing (ideas: list_tables(), describe_table(name), make_line_chart(...)). Register them with the Anthropic SDK using the tool definitions schema, and implement the conversation loop so the LLM can call them in sequence.

Test with a multi-step question that requires at least two tool calls: for example, “List the available tables, then show me the top 5 countries by revenue as a bar chart.” Inspect the conversation history and verify you can see each tool call and its result.

Exercise 8: Sandbox a Code Execution Tool

Add a run_python(code: str) -> str tool to your agent from Exercise 7. The tool should execute the provided Python code and return stdout.

Run this version first with no sandboxing and confirm it works. Then sandbox it: run the generated code inside a Docker container with --network none, a memory cap, and --read-only. Pass the database query results in as a CSV string in the code, so the container never needs a database connection.

Compare the output of the sandboxed and unsandboxed versions on the same question. What limitations does the sandbox impose? What security risks does it eliminate?
