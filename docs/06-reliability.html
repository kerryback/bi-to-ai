6  Reliability and Compliance

The previous five modules built an increasingly powerful toolkit: agent concepts, coding tools, data analysis, thinking-partner conversations, and database skills. The next two modules will add agent workflows and document-based AI. Before any of this is deployed, leadership will ask one question: can we trust it?

This module provides the answer, which is yes — with the right verification. You will learn a repeatable protocol for validating AI outputs, stress-testing skills and RAG systems, and building the governance framework your organization needs.

6.1 The trust spectrum

AI with code execution is like a fast junior analyst — capable but in need of oversight. The appropriate level of oversight depends on the stakes.

For low-stakes work (exploratory charts, directional trends, internal brainstorming), a quick sanity check is sufficient. Glance at the numbers, confirm they are in the right ballpark, and move on.

For medium-stakes work (team skills from Module 5, recurring reports, RAG-based Q&A from Module 8), test with known answers before relying on the results. Run queries where you already know the answer and confirm that AI matches.

For high-stakes work (board presentations, budget recommendations, external reporting), apply the full verification protocol described later in this module. Every number, every chart, every claim should be independently verified.

Key Terms
  • Hallucination — AI generating plausible but incorrect output (wrong numbers, fabricated citations, confident but false claims).
  • RAG (Retrieval-Augmented Generation) — grounding LLM responses in retrieved documents so answers cite real sources rather than relying on the model’s training data alone.
  • Red-teaming — deliberately attacking your own system to find weaknesses before real users or adversaries do.1
  • Guardrails — technical constraints that prevent misuse: read-only database access, restricted tables, input validation, and query logging.
  • Defense in depth — layering multiple independent safeguards so that no single failure compromises the system.
  • PII (Personally Identifiable Information) — data that can identify an individual (names, Social Security numbers, email addresses, etc.).
  • Data sovereignty — the legal requirement that data remain under a specific jurisdiction’s control.2
  • Air-gapped — infrastructure with no internet connectivity, used for classified or highly sensitive workloads.

6.2 Hallucination: the core risk

“How do I know the AI isn’t making things up?”

AI in code-execution mode does not invent facts the way a chatbot might. It reads real data and runs real computations. But it makes silent analytical errors: wrong filters, dropped rows, incorrect joins, misinterpreted column names. In RAG mode, it can also hallucinate citations — citing a page that does not actually contain the claim.

These errors are dangerous precisely because the output looks professional. Charts render cleanly even when the underlying data is wrong. Memos read smoothly even when the conclusion does not follow from the evidence. A polished result inspires confidence, and confidence without verification is the real risk.

You have already seen examples. Date or currency parsing surprises in Module 3. Incorrect join paths in Module 5. Module 8 will show how RAG can produce hallucinated citations. Each error is fixable, but only if someone checks.

6.3 The verification protocol

A three-step process you can apply to any AI-generated analysis.

The first step is to sanity-check with known answers. Ask questions where you already know the result: row count, column names, a total you can verify in Excel. “How many rows are in this dataset?” “What is the total revenue?” If AI’s answer does not match your independently verified number, stop and investigate before proceeding.

The second step is to inspect the methodology. Ask “show me the code” or “show me the SQL.” Read the logic, not just the output. Check that the joins are correct, the filters make sense, and the aggregation matches your intent. Then rephrase the question and run it again. If the answers agree, you have confirmation. If they differ, one approach is wrong.

The third step is to spot-check edge cases. Test the smallest group, zero values, null fields, and boundary dates. Errors hide at the edges. A query that works perfectly for the top ten countries may break for the bottom three.

If you would spot-check a human analyst’s work, spot-check AI’s.

6.4 Catching silent errors

To make this concrete, consider a deliberately tricky prompt.

“What’s the average profit margin by sub-category? Exclude any sub-categories with negative total profit.”

The trap is in the filter logic. Does AI filter before or after aggregating? Does it drop individual rows with negative profit, or sub-categories whose total profit is negative? The order matters, and AI often gets it wrong.

To catch it, ask three follow-up questions. “Show me the code” — inspect the filter logic. “How many rows are in the result?” — does the count match your expectation? “List the excluded sub-categories” — are they the right ones?

The more specific the filter logic, the more likely AI misinterprets it.

6.5 Trusting RAG answers

RAG (Module 8) grounds AI in your documents, but grounding does not guarantee correctness.

RAG introduces three specific risks. The first is hallucinated citations: the LLM cites a page or section that does not actually contain the claim. The citation looks authoritative, but when you check the source, the passage says something different or does not address the question at all.

The second is stale documents. If a policy changed but the old version is still in the index, RAG will retrieve and cite the outdated version. The answer is grounded — just in the wrong version of the truth.

The third is wrong-chunk retrieval. The retrieved passage is semantically similar to the question but does not actually answer it. This happens when the document contains multiple sections that use similar language for different purposes.

The verification approach is the same: click through every citation and confirm that the cited passage actually says what the LLM claims. Check document dates to ensure you are getting the current version. Ask the same question in different words to see if the citation changes.

A citation that looks right but is not is worse than no citation at all. It creates false confidence.

6.6 Red-teaming RAG

Red-teaming means deliberately attacking your own system to find weaknesses before real users do.3 Test your RAG system the way an adversary would.

Ask out-of-scope questions: “What is our competitor’s revenue?” when no competitor data is in the documents. Does AI admit it does not know, or does it hallucinate an answer?

Upload two documents that disagree on a fact and ask about the disputed point. Does AI cite both? Pick one? Acknowledge the conflict?

Ask boundary questions about topics at the edge of what is documented. These are the questions most likely to produce hallucinated citations, because the system retrieves a passage that is close but not quite on point.

Every failure you find now is one your team will not encounter in production.

6.7 Red-teaming your skill

The database skill from Module 5 is the kind of tool people stop questioning once it is deployed. Test it like any tool your team depends on.4

Validate with known answers by running queries where you already know the result (total invoices, number of customers). Compare AI’s SQL to a hand-written query. Ask the same question multiple times and check for consistency.

Stress-test with edge cases. Ask for revenue from a nonexistent country. Query boundary dates (the first and last month in the dataset). Use ambiguous column names (“show me the name” when multiple tables have a name column).

For each failure, diagnose the root cause (bad SQL, missing context, or wrong table), fix the SKILL.md, and retest. This is how production skills get hardened. Every failure makes the skill better.

6.8 Three layers of auditability

Production AI systems should have multiple layers of verification.

The first layer is RAG: the LLM answers based on your documents, with citations. This grounds responses in source material. You verify by checking citations against the original text.

The second layer is code transparency: the agent displays the SQL or Python it ran. Analysts review the logic, not just the answer. This is what you practiced in Modules 3 through 5.

The third layer is guardrails: read-only database access, restricted tables, and every query logged for audit. The system prevents mistakes, not just detects them.

The more layers you add, the harder it is for errors to reach your team. Defense in depth — the same principle as cybersecurity.

6.9 Security and governance

Before deploying AI with your data, three questions need clear answers: what data leaves your network, where does inference happen, and what audit trail exists?

Cloud API

When you call a cloud API — OpenAI, Anthropic, Google — your application sends a prompt to the provider’s shared infrastructure, the LLM runs inference there, and the response comes back. The key question is what is in that prompt.

As Module 5 showed, you often need to send only a schema description and a user question. The LLM returns SQL or code, and your application executes it locally against your own database. The actual data — customer names, revenue figures, health records — never leaves your network. In this pattern, cloud inference is compatible with most data policies because the sensitive data stays behind your firewall.

However, some use cases do require sending data to the LLM. RAG prompts include document chunks. Summarization tasks send the full text to be summarized. In these cases, the data does travel to the provider’s servers, and the provider’s data-processing agreement governs what happens to it.

Enterprise API tiers from Anthropic, OpenAI, and Google typically guarantee that your data is not used for training, offer short or zero retention windows, and provide contractual commitments on data handling. Consumer tiers (free ChatGPT, free Claude) may not offer these guarantees. Always confirm the terms before sending sensitive data.

Dedicated cloud tenancy

Dedicated cloud tenancy means running inference on isolated instances within a public cloud provider — hardware in the provider’s data center but reserved exclusively for your organization and walled off from other tenants.5 Examples include AWS Dedicated Hosts, Azure Dedicated Host, and GCP sole-tenant nodes.6

For closed-source models, the providers offer dedicated deployment options. Anthropic offers Claude on Amazon Bedrock and Google Cloud Vertex AI, where inference runs in your dedicated tenancy under your security controls. OpenAI offers Azure OpenAI Service, where GPT models run in your Azure subscription. These are enterprise licensing arrangements: you pay for compute and per-token usage, and the provider guarantees that your data stays within your tenancy and is not used for training.

Dedicated tenancy gives you the security isolation of on-premise infrastructure with the operational convenience of a cloud provider. It is a common choice for organizations that need strong data controls but do not want to manage their own GPU hardware.

Private cloud and on-premise inference

Private cloud means running the LLM on servers your organization owns or operates — in your own data center, with cloud-like characteristics such as elastic scaling and self-service provisioning.7 Examples include HPE GreenLake and Dell APEX. On-premise deployments without these cloud characteristics are simply traditional data-center hosting. In either case, nothing leaves your network. You control the hardware, the model weights, and the logs.

For open-source models (Llama, Mistral, DeepSeek), you download the model weights and run inference on your own GPU servers. There is no vendor relationship for the model itself — you manage everything.

On-premise and private-cloud deployments give maximum control but require GPU infrastructure and operational expertise. They may also lag behind the latest cloud-hosted models in capability.

Air-gapped environments — on-premise infrastructure with no internet connectivity — represent the most restrictive deployment. These are required for classified and military workloads where even outbound network traffic is prohibited.

Choosing the right deployment

Most organizations need a mix. Cloud APIs work well for non-sensitive tasks where only schema or public information is sent. Dedicated cloud tenancy suits workloads that need isolation without the burden of managing hardware. Private cloud or on-premise deployment is appropriate for the most sensitive workloads: regulated records, proprietary documents, or data subject to strict sovereignty requirements.

The deciding factor is not the model but the data. Ask: does this prompt contain information that cannot leave our network? If yes, run inference in a dedicated tenancy or on-premise. If no, a cloud API with an enterprise data-processing agreement is typically sufficient.

Data sovereignty

Data sovereignty is the legal and organizational requirement that data remain under the jurisdiction and governance controls of a specific country or entity.8 It goes beyond the cloud-vs.-on-prem deployment choice: even data stored in a cloud provider’s data center may violate sovereignty requirements if that data center is in another country or subject to another government’s laws.

GDPR requires that EU residents’ personal data be processed under EU-adequate protections. U.S. defense agencies require air-gapped environments — infrastructure with no internet connectivity — for classified workloads. Financial regulators in many countries require that customer records stay within national borders. When you connect an AI agent to enterprise data, these constraints determine not just how inference runs but where it runs.

The pattern emerging across industries is hybrid: sensitive and regulated workloads run on private infrastructure (on-premises or sovereign cloud), while non-sensitive tasks use public cloud APIs. The U.S. Department of Defense awarded a $931 million contract to modernize its data centers with on-premise private cloud. Barclays migrated over 50,000 workloads to private cloud infrastructure for regulatory compliance. New Zealand telecom provider 2degrees built a sovereign AI platform to ensure customer data stays under New Zealand jurisdiction. These are not edge cases — they reflect the standard enterprise pattern for AI deployment with sensitive data.

Audit requirements

SOX, GDPR, HIPAA, and industry-specific regulations all impose requirements on how data is processed and by whom.9 Every prompt sent to an LLM is a data-processing event and must be treated accordingly — logged, governed, and reviewed.

6.10 Data policies: training and retention

Before sending data to any AI provider, know the answers to two questions.

Will they train on your data? Consumer tiers (free ChatGPT, free Claude) may use your input to improve their models. API and enterprise tiers typically do not. Check your agreement and confirm that any opt-out is active.

How long do they retain it? Providers retain prompts for a window (often 30 days) for safety monitoring. Enterprise contracts may offer zero retention. Know the policy before you send sensitive data.

Ask your vendor three things: Do you train on our data? How long do you retain it? Can we request deletion?

6.11 Compliance in practice

FERPA protects student education records. HIPAA protects health information. SOX governs financial reporting. GDPR regulates personal data in the EU. Every regulated industry has its version of the same concern: where does the data go, and who can see it?

AI can analyze sensitive data without ever reading the actual records. The technique from Module 5 applies: describe the table structure in the SKILL.md (table names, column names, data types), and the agent writes SQL from the schema alone, never seeing the actual records. The script runs locally, and the output stays on your server.

Table metadata is not PII. The actual records stay behind the firewall. The LLM produces the code, not the output. This pattern — send the schema, keep the data — works for any regulated environment.

You can improve the quality of the generated SQL by including a few synthetic rows alongside the schema. Databricks found that adding approximately three sample rows per table raises SQL accuracy by about six percentage points on standard benchmarks compared to schema alone. The sample rows reveal data formatting patterns (is the status column stored as Active/Inactive or 1/0?), disambiguate column semantics (is code a country code or a product code?), and help the model write correct WHERE clauses with properly formatted literal values. The key is that the rows are synthetic — fabricated to match the real data’s formats and value ranges without containing any actual records. Three rows is the sweet spot; more than that tends to distract the model rather than help it. Keep the synthetic data in sync with your schema: when you add a column or change a data type in production, update the synthetic rows to match.

6.12 Session log auditing

Every AI conversation can be exported as a transcript. The transcript contains the user’s questions, the AI’s responses, the code it generated, and the results it produced. This is the paper trail your compliance team needs.

Review session logs for three things: PII or proprietary data that should stay internal, SQL or Python logic errors in the generated code, and RAG citations that do not match the source.

Export logs, archive them, and review them periodically. Session logs are not just a compliance requirement; they are a learning tool. Reviewing what went wrong in past sessions tells you exactly what to improve in your SKILL.md or RAG configuration.

6.13 The sandbox-audit-deploy pattern

Before deploying any AI workflow to your team, follow three phases.

In the sandbox phase, run the full workflow with synthetic or test data. Test edge cases. Break it on purpose. This is where you discover the failure modes described throughout this module.

In the audit phase, export session logs and review the generated code. Verify SQL logic. Check RAG citations. Look for data leakage. Have a colleague who did not build the system try to break it.

In the deploy phase, roll out to users with monitoring. Review logs weekly. Set up alerting for anomalies. Treat the first month as a soft launch.

6.14 The adoption roadmap

Each stage of AI adoption maps to a trust level.

Quick wins, achievable in weeks, require only light verification. Using AI for research, summarization, and email drafting. Analyzing local data with a coding agent (Module 3). Having thinking-partner conversations (Module 4).

Medium-term adoption, taking months, requires test-before-deploy discipline. Skills for recurring database workflows (Module 5). Agent-powered reports and pipelines (Module 7). RAG for document-based Q&A (Module 8).

Strategic adoption, measured in quarters, requires a full audit trail. Production apps with authentication and audit trails. An AI usage policy, compliance review, and governance framework. Organization-wide adoption with monitoring.

6.15 Exercises

Exercise 1: The Verification Protocol

Run the complete three-step verification protocol on a real analysis.

First, sanity-check: open the Superstore data in your coding agent and ask three questions where you know the answer (row count, total revenue, number of unique customers). Does AI match?

Second, inspect methodology: for three queries, ask “show me the code.” Read each script. Is the logic correct? Then rephrase each question and run it again. Do the answers agree?

Third, spot-check edge cases: test three edge cases (empty results, boundary dates, null values). Does AI handle them gracefully?

Exercise 2: Red-Team Your Skill

Open your coding agent with the Chinook database and your SKILL.md from Module 5.

Run five adversarial queries designed to break the skill: a query about nonexistent data (“Sales in 2025”), a complex multi-table join (“Revenue by artist by country”), a request for data not in the schema (“Customer satisfaction scores”), an ambiguous column name, and a boundary date query.

For each failure, diagnose the root cause, fix the SKILL.md, and retest.

Exercise 3: RAG Citation Audit

Open your coding agent with the company handbook from Module 8 and ask five factual questions.

For each answer, locate every cited passage in the original document. Rate each citation as accurate, partially accurate, or hallucinated.

Then ask three questions the document cannot answer. Does AI admit it does not know, or does it hallucinate?

Exercise 4: Draft an AI Usage Policy

Write a one-page AI usage policy for your organization (or a hypothetical one).

Cover four areas. First, what data can and cannot be sent to AI providers, classified by sensitivity level. Second, which AI tools are approved for which use cases. Third, when human review is required before acting on AI output. Fourth, how RAG citations should be verified before sharing results externally.

Exercise 5: Compliance Scenario Analysis

Identify the primary data regulation in your industry (HIPAA, SOX, GDPR, PCI-DSS, FERPA, or another).

Describe a specific use case: what data would AI access, and what questions would it answer? Design the compliant workflow: what information goes to AI (schema only, or actual data)? What stays local? Where are the audit points?

List three guardrails you would add to enforce compliance.
